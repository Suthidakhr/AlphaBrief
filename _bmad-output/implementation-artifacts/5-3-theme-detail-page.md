# Story 5.3: Theme Detail Page (`/trends/[id]`)

**Status:** done

## Story

As a retail investor,
I want to open a theme and read all its constituent articles with individual AI analysis and a compliance disclaimer,
So that I can understand the full evidence base for a market theme before forming a view.

## Acceptance Criteria

### AC1 — Page content order with valid, active theme

**Given** `/trends/[id]` loads with a valid, active theme ID
**When** the page renders
**Then** content order top-to-bottom is:
1. Theme name as `<h1>` (20px/`text-xl`, 700/`font-bold`, espresso `#4A342A`) + `SentimentBadge` in the same row
2. Article count + date range of constituent articles (cocoa `#7D5A44`, `text-sm`)
3. Theme description (cocoa `#7D5A44`, `text-sm leading-relaxed`)
4. Constituent articles rendered as `NewsCard` components sorted by `published_at` descending
5. Non-removable AI disclaimer (FR-A04): "This analysis is generated by AI for educational purposes only. It does not constitute investment advice. Always consult a qualified financial advisor before making investment decisions."

**And** the disclaimer is a structural part of the page — not a prop, not conditional, not a link to `/about`

### AC2 — Constituent articles render as NewsCard

**Given** a constituent article inside a `MarketTheme`
**When** it renders on the Theme Detail page
**Then** it renders using the existing `<NewsCard>` component
**And** `AIInsightBox` renders inside `NewsCard` — if `analysis_at` is > 24h old, the amber staleness indicator appears automatically (same `AIInsightBox` component handles this, no Theme-Detail-specific staleness logic needed)
**And** if `ai_analysis` is `null`, `AIInsightBox` shows the "Analysis in progress" pending state (cocoa, animated dot)

### AC3 — HTTP 410 (theme expired) state

**Given** `/trends/[id]` where `last_article_at` is older than 48 hours (API returns `HTTP 410`)
**When** the page renders
**Then** it shows: "This theme has expired. Themes with no new articles in 48 hours are archived."
**And** a link to `/trends` is present with label "Browse current themes"
**And** it does NOT throw a Next.js error page — Navbar, footer remain functional
**And** this state is detected by parsing `err.message.includes("410")` (same pattern as 404 detection in `NewsDetailPage`)

### AC4 — HTTP 404 (non-existent theme) state

**Given** `/trends/[id]` with a non-existent ID (API returns `HTTP 404`)
**When** the page renders
**Then** it shows: "Theme not found."
**And** a link to `/trends` is present with label "Browse current themes"
**And** it does NOT use Next.js `notFound()` — renders inline error message, Navbar/footer remain functional

### AC5 — Suspense boundary and loading skeleton

**Given** a cold load with no ISR cache
**When** Suspense boundaries resolve
**Then** 3× `<SkeletonCard />` components render as fallback — no blank white regions
**And** the existing `SkeletonCard` component is reused (`frontend/src/components/SkeletonCard.tsx`) — do NOT create a new skeleton

### AC6 — Empty constituent articles state

**Given** a `MarketTheme` with `constituent_articles: []` (theoretically impossible by design but guarded)
**When** it renders
**Then** it shows: "No articles available for this theme."

### AC7 — Back-link and page layout

**Given** the `/trends/[id]` page
**When** it renders
**Then** there is a linen (`#F5F1EA`) breadcrumb bar below `<TickerBar />` with a `← Market Trends` link navigating to `/trends`
**And** the main content area uses `max-w-3xl mx-auto px-4 py-6` (same as `/news/[id]`)
**And** the page has `<Navbar />`, `<TickerBar />`, breadcrumb bar, `<main>`, footer — in that order
**And** `<main>` has `id="main-content"`

## Dev Notes

### Architecture: What already exists — use it, don't reinvent

**`api.getTheme(id)` is already implemented** (`frontend/src/lib/api.ts:35`):
```ts
getTheme: (id: string) => fetchAPI<MarketTheme>(`/trends/${id}`),
```
`fetchAPI` throws `Error` with message `"API error: ${res.status} ${path}"` on any non-ok response.

**`MarketTheme` type already exists** (`frontend/src/types/index.ts:102`):
```ts
export interface MarketTheme extends MarketThemeSummary {
  constituent_articles: NewsItem[];
}
```
`MarketThemeSummary` has: `theme_id`, `name`, `description`, `overall_sentiment`, `article_count`, `last_article_at`, `created_at`.
`NewsItem` has: `id`, `headline`, `summary`, `source_url`, `content`, `category`, `published_at`, `source`, `ai_analysis: AIAnalysis | null`, `stock_impacts`, `featured`.

**Existing components to reuse (no modifications needed):**

### Pattern: Dynamic route page

Copy the structure from `frontend/src/app/news/[id]/page.tsx`:
```tsx
export default async function ThemeDetailPage({
  params,
}: {
  params: Promise<{ id: string }>;
}) {
  const { id } = await params;
  // ...
}
```
This is Next.js 14 App Router — `params` is `Promise<{ id: string }>`, must be awaited.

### Pattern: Inner server function

Use the `ThemeDetailServer` inner async function pattern — same as `ThemesServer` in `trends/page.tsx` and `NewsDetailServer` in `news/[id]/page.tsx`. The inner function isolates data-fetch errors so Navbar/TickerBar/footer remain functional if the theme fetch fails.

### Critical: HTTP 410 vs 404 error handling — do NOT use `notFound()`

`fetchAPI` error messages follow the pattern: `"API error: 410 /trends/${id}"`. Parse with:
```tsx
if (err instanceof Error && err.message.includes("410")) {
  // render "This theme has expired..." inline JSX
}
if (err instanceof Error && err.message.includes("404")) {
  // render "Theme not found." inline JSX
}
// otherwise re-throw so Next.js error boundary catches
throw err;
```
**NEVER call `notFound()` here** — the AC says both 410 and 404 must NOT throw Next.js error pages. Inline JSX only.

The pattern from `NewsDetailPage` calls `notFound()` for 404 — **this story intentionally diverges** because the theme detail error states need to show explanatory text with a link back to `/trends`.

### Date range derivation

`MarketTheme` has no explicit date range field. Derive from `constituent_articles`:
```tsx
function dateRange(articles: NewsItem[]): string {
  if (articles.length === 0) return "";
  const dates = articles.map((a) => new Date(a.published_at).getTime());
  const oldest = new Date(Math.min(...dates));
  const newest = new Date(Math.max(...dates));
  const fmt = (d: Date) =>
    d.toLocaleDateString("en-US", { month: "short", day: "numeric", year: "numeric" });
  return oldest.getTime() === newest.getTime()
    ? fmt(oldest)
    : `${fmt(oldest)} – ${fmt(newest)}`;
}
```
Render in AC1 item 2 as: `{theme.article_count} {theme.article_count === 1 ? "article" : "articles"} · {dateRange(theme.constituent_articles)}`
